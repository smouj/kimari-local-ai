# First Private SFT Handoff — Bringing Sanitized Results to the Repository

> **Document Type:** Handoff guide for transferring sanitized artifacts from RunPod / local GPU to the Kimari Local AI repository  
> **Version:** v0.1.28-alpha  
> **Date:** 2026-05-30  
> **Status:** Active — governs what may leave the training environment and enter the repo

---

## 1. Purpose

After a private SFT training run completes on RunPod or a local GPU, you will have a directory full of artifacts — adapter weights, eval outputs, manifests, configs, screenshots, and more. **Almost none of it can enter the repository as-is.**

This document defines the handoff process: how to select, sanitize, review, and commit only the artifacts that are safe for the repo. Its purpose is threefold:

1. **Prevent secrets leakage.** API keys, HuggingFace tokens, absolute filesystem paths, and private user data must never enter git history. Once pushed, they are effectively public — even if deleted in a later commit.

2. **Prevent weight/asset leakage.** Adapter weights, checkpoints, GGUF exports, and raw model outputs are large, proprietary, and governed by the `ADAPTER_PREVIEW_GATE`. They are never committed.

3. **Preserve traceability.** A carefully sanitized subset of metadata — the manifest, eval summaries, comparison summaries, and run record — provides a durable audit trail proving the run happened and describing what it produced, without exposing sensitive content.

### What Can Enter the Repo

Only **sanitized metadata artifacts** that have been individually reviewed and confirmed clean:

| Artifact | Sanitization Method |
|----------|-------------------|
| `MANIFEST.yaml` | Manual review — remove private paths, secrets; confirm `preview_gate_state: BLOCKED` |
| `eval_summary.json` | Generated by `create_eval_summary.py` — strips prompts, responses, and local paths |
| `compare_summary.json` | Sanitized — no private prompt content, no raw responses, only category-level aggregates |
| Private run record | Generated by `create_private_run_record.py` — strips private paths, no secrets |
| Screenshots | Reviewed following `SAFE_SCREENSHOT_CAPTURE.md` |

### What Can NEVER Enter the Repo

| Artifact | Why |
|----------|-----|
| Adapter weights (`.safetensors`, `.bin`) | Proprietary binary data — governed by `ADAPTER_ARTIFACT_POLICY.md` |
| Training checkpoints (`checkpoint-*/`) | Intermediate optimizer states and weight snapshots |
| GGUF exports (`*.gguf`) | Quantized model binaries |
| Raw eval outputs (`*-raw.json`) | Contains full model responses with potentially private prompts |
| Optimizer states (`.pt`, `.pth`) | Training state only — not useful, potentially revealing |
| Merged model files | Full model weights — very large, proprietary |

> **Core principle:** When in doubt, leave it out. If you are unsure whether an artifact can be committed, do not commit it. Seek review first.

## 2. Before Handoff — Security Checklist

Complete **every item** on this checklist before transferring any artifact from the training machine to the repository. This applies whether you are on RunPod, a local GPU, or any other environment.

### 2.1 Verify Training Is Complete

- [ ] Training run has finished (no running processes)
- [ ] Post-run script has been executed: `python training/scripts/postrun_private_sft.py --dry-run`
- [ ] All expected output files exist in `training/adapters/<run_id>/`

### 2.2 Confirm No Secrets in the Training Environment

- [ ] No `HF_TOKEN` or `hf_` strings in any output file
- [ ] No API keys, passwords, or authentication tokens in any output file
- [ ] No `.env` files or secret configuration in the adapter directory

> See [`docs/HF_TOKEN_SAFETY.md`](HF_TOKEN_SAFETY.md) for HuggingFace token handling procedures.

### 2.3 Confirm No Absolute Paths in Output Files

RunPod writes absolute paths like `/workspace/kimari-local-ai/...`. Local GPUs write paths like `/home/username/...`. These must be stripped before any file leaves the training machine.

```bash
# Check for RunPod absolute paths
rg '/workspace/' training/adapters/kimari-smollm3-sft-v0/ 2>/dev/null

# Check for home directory absolute paths
rg '/home/[a-z]' training/adapters/kimari-smollm3-sft-v0/ 2>/dev/null

# Check eval results for paths
rg '/workspace/' eval/results/ 2>/dev/null
rg '/home/[a-z]' eval/results/ 2>/dev/null
```

### 2.4 Confirm Gate State Is BLOCKED

- [ ] `MANIFEST.yaml` has `preview_gate_state: BLOCKED`
- [ ] `MANIFEST.yaml` has `hf_upload_allowed: false`
- [ ] `MANIFEST.yaml` has `public_release_allowed: false`

> The preview gate **never** advances as a side effect of handoff. See [`docs/ADAPTER_ARTIFACT_POLICY.md`](ADAPTER_ARTIFACT_POLICY.md).

### 2.5 Confirm No Weight Files in Staging

```bash
# From the repo root on the local machine
git status | rg '\.safetensors'
git status | rg '\.bin'
git status | rg '\.pt'
git status | rg '\.gguf'
git status | rg 'checkpoint-'
```

All of these should return nothing. If they match any file, **stop** — that file must not be committed.

## 3. What Can Be Committed — Sanitized Artifacts with Review Requirements

Each artifact below has specific sanitization requirements. **Do not commit any artifact that has not passed its sanitization step.**

### 3.1 MANIFEST.yaml — Reviewed Manifest

**What it is:** The adapter manifest recording metadata about the training run — adapter name, base model, LoRA config, training date, hashes, and gate state.

**Sanitization steps:**

```bash
# 1. Copy manifest to a staging area (do not edit the original)
cp training/adapters/kimari-smollm3-sft-v0/MANIFEST.yaml /tmp/MANIFEST_staged.yaml

# 2. Remove all absolute paths
sed -i 's|/workspace/kimari-local-ai/||g' /tmp/MANIFEST_staged.yaml
sed -i 's|/home/[^/]*/kimari-local-ai/||g' /tmp/MANIFEST_staged.yaml

# 3. Verify no sensitive content remains
rg '/workspace' /tmp/MANIFEST_staged.yaml   # Must find nothing
rg '/home/' /tmp/MANIFEST_staged.yaml       # Must find nothing
rg 'hf_' /tmp/MANIFEST_staged.yaml          # Must find no tokens
rg 'secret' /tmp/MANIFEST_staged.yaml       # Must find no secrets
```

**Review requirements:**

- [ ] No absolute paths (`/workspace/...`, `/home/...`)
- [ ] No HF tokens or API keys
- [ ] No weight data or binary content
- [ ] `preview_gate_state: BLOCKED`
- [ ] `hf_upload_allowed: false`
- [ ] `public_release_allowed: false`
- [ ] All paths are repo-relative (e.g., `training/configs/...` not `/workspace/.../training/configs/...`)

**Then copy the sanitized manifest into the repo:**

```bash
cp /tmp/MANIFEST_staged.yaml training/adapters/kimari-smollm3-sft-v0/MANIFEST.yaml
```

> **Note:** If the `training/adapters/` directory is gitignored, you may need to use `git add -f` or commit the manifest to a different location. See [`docs/ADAPTER_ARTIFACT_POLICY.md`](ADAPTER_ARTIFACT_POLICY.md) for guidance.

### 3.2 eval_summary.json — Sanitized via create_eval_summary.py

**What it is:** An aggregate summary of evaluation results — run ID, model label, prompt count, category counts, score status, safety flags. **No prompt text. No response text. No local paths.**

**Sanitization is automatic** — use the dedicated script:

```bash
python eval/scripts/create_eval_summary.py \
    --input eval/results/adapter-smollm3-sft-v0-q4km.json \
    --output eval/results/adapter-smollm3-sft-v0-q4km-summary.json
```

The script strips all `prompt` and `response` fields and produces only safe aggregate metadata.

**Review requirements:**

- [ ] Generated by `create_eval_summary.py` (not manually edited)
- [ ] Contains only: `run_id`, `model_label`, `kimari_version`, `prompt_count`, `category_counts`, `score_status`, `manual_review_required`, `safety_regression_detected`
- [ ] No `"prompt"` or `"response"` keys with actual content
- [ ] No absolute filesystem paths
- [ ] `score_status` is `"manual_review_required"` (never invented)
- [ ] No `hf_` strings or API keys

> The raw eval output file (`adapter-smollm3-sft-v0-q4km.json`) must **never** be committed. See [`docs/PRIVATE_RUN_ARTIFACTS.md`](PRIVATE_RUN_ARTIFACTS.md).

### 3.3 compare_summary.json — Sanitized Comparison Summary

**What it is:** A category-level comparison of adapter vs. baseline results — "improved," "regressed," or "unchanged" per category, plus overall improvement and safety regression flags. **No private prompt content. No raw model responses.**

**Sanitization steps:**

```bash
# 1. If a script generates the comparison, use it
python eval/scripts/compare_runs.py \
    --baseline eval/results/baseline-smollm3-q4km.json \
    --adapter eval/results/adapter-smollm3-sft-v0-q4km.json \
    --summary-output eval/results/comparison-sft-v0-vs-baseline-summary.json

# 2. Manual review — open the file and verify
#    - Only category-level aggregates present
#    - No "prompt" or "response" fields with content
#    - No absolute paths
#    - No HF tokens or secrets
```

**Review requirements:**

- [ ] Contains only category-level aggregates (improved / regressed / unchanged)
- [ ] No private prompt content
- [ ] No raw model response text
- [ ] No absolute filesystem paths
- [ ] No `hf_` strings or API keys
- [ ] Overall improvement flag and safety regression flag are present
- [ ] No invented scores or fabricated comparisons

### 3.4 Private Run Record — via create_private_run_record.py

**What it is:** A structured JSON record that ties together the run's identity, config, and artifact references. It proves the run happened and describes it — without distributing outputs.

**Sanitization is built in** — the script rejects absolute home directory paths and always sets `preview_gate_state: BLOCKED`:

```bash
python training/scripts/create_private_run_record.py \
    --run-config training/configs/private_sft_run.v0.yaml \
    --manifest training/adapters/kimari-smollm3-sft-v0/MANIFEST.yaml \
    --eval-summary eval/results/adapter-smollm3-sft-v0-q4km-summary.json \
    --compare-summary eval/results/comparison-sft-v0-vs-baseline-summary.json \
    --output /tmp/private_run_record.json
```

Preview with `--dry-run` first:

```bash
python training/scripts/create_private_run_record.py \
    --run-config training/configs/private_sft_run.v0.yaml \
    --dry-run
```

**Review requirements:**

- [ ] Generated by `create_private_run_record.py`
- [ ] `preview_gate_state: BLOCKED`
- [ ] `public_release_allowed: false`
- [ ] `hf_upload_allowed: false`
- [ ] No absolute home directory paths (the script rejects these)
- [ ] No `hf_` strings or API keys
- [ ] No weight data, no raw eval outputs

> See [`docs/FIRST_PRIVATE_SFT_RECORD.md`](FIRST_PRIVATE_SFT_RECORD.md) for the full run record specification.

### 3.5 Reviewed Screenshots

**What they are:** Terminal or UI screenshots documenting the training run, evaluation, or CLI output.

**Screenshots must follow** [`docs/SAFE_SCREENSHOT_CAPTURE.md`](SAFE_SCREENSHOT_CAPTURE.md) in its entirety. Key requirements:

- [ ] No visible secrets, tokens, or API keys
- [ ] No private user paths (`/home/realname/`)
- [ ] No raw eval outputs or unreviewed benchmark scores
- [ ] Kimari-4B is not presented as published or available
- [ ] Filename follows convention: `kimari-<command>.png`
- [ ] Image has been reviewed at full resolution before committing
- [ ] Compressed with `oxipng` or `optipng` to minimize repo bloat

> **Screenshots are irreversible.** Once a PNG enters git history, it persists even if deleted in a later commit. Treat every screenshot commit as permanent.

## 4. What Must Remain Local — Heavy/Sensitive Outputs That Never Enter the Repo

The following artifacts must **never** leave the training machine or be committed to any repository. This list is non-exhaustive — when in doubt, keep it local.

| Artifact | Location Pattern | Why It Must Stay Local |
|----------|-----------------|----------------------|
| **LoRA adapter weights** | `training/adapters/**/*.safetensors` | Proprietary binary weight data — not for distribution |
| **Legacy weight files** | `training/adapters/**/*.bin` | Same concerns as safetensors |
| **Training checkpoints** | `training/adapters/**/checkpoint-*/` | Intermediate optimizer states and weight snapshots |
| **Optimizer states** | `training/adapters/**/*.pt` | `optimizer.pt`, `rng_state.pth`, `scheduler.pt` — training state only |
| **Merged model files** | `training/adapters/*-merged/` | Full model weights after adapter merge — very large |
| **GGUF exports** | `models/kimari-*.gguf` | Quantized model files for inference — large binary |
| **Raw eval outputs** | `eval/results/*-raw.json` | Contains full model responses with potentially private prompts |
| **Raw comparison outputs** | `eval/results/comparison-*-raw.json` | May contain prompt text or response text |
| **TensorBoard logs** | `training/adapters/**/runs/` | Binary event files, not useful in Git |
| **WandB local cache** | `wandb/` | May contain experiment data or API keys |
| **Training data copies** | `training/adapters/**/*.jsonl` | License restrictions; data should come from `dataset/build/` |
| **Heavy log files** | `training/adapters/**/*.log` | Potentially large, may contain sensitive information |
| **Base model weights** | HuggingFace cache or download directory | Base model weights are never committed — downloaded at training time |

> **Backup locally.** These artifacts should be backed up to private storage (external drive, encrypted cloud) but must never enter the Git repository. See [`docs/ADAPTER_ARTIFACT_POLICY.md`](ADAPTER_ARTIFACT_POLICY.md) for backup procedures.

## 5. Handoff Process — Step-by-Step

Follow these steps in order. **Do not skip any step.** Each step builds on the previous one.

### Step 1: Complete the Training Run

Ensure the training run has fully completed and the post-run script has been executed:

```bash
# Verify training is done
ps aux | rg "python.*train"   # Should find nothing

# Run post-training orchestration (dry-run first)
python training/scripts/postrun_private_sft.py --dry-run

# Then execute for real
python training/scripts/postrun_private_sft.py \
    --run-config training/configs/private_sft_run.v0.yaml \
    --adapter-dir training/adapters/kimari-smollm3-sft-v0
```

### Step 2: Generate Sanitized Summaries on the Training Machine

Run each sanitization script on the training machine **before** transferring any files:

```bash
# Generate eval summary (strips prompts, responses, paths)
python eval/scripts/create_eval_summary.py \
    --input eval/results/adapter-smollm3-sft-v0-q4km.json \
    --output eval/results/adapter-smollm3-sft-v0-q4km-summary.json

# Generate compare summary (if baseline eval exists)
python eval/scripts/compare_runs.py \
    --baseline eval/results/baseline-smollm3-q4km.json \
    --adapter eval/results/adapter-smollm3-sft-v0-q4km.json \
    --summary-output eval/results/comparison-sft-v0-vs-baseline-summary.json

# Generate private run record (rejects absolute home paths)
python training/scripts/create_private_run_record.py \
    --run-config training/configs/private_sft_run.v0.yaml \
    --manifest training/adapters/kimari-smollm3-sft-v0/MANIFEST.yaml \
    --eval-summary eval/results/adapter-smollm3-sft-v0-q4km-summary.json \
    --compare-summary eval/results/comparison-sft-v0-vs-baseline-summary.json \
    --output /tmp/private_run_record.json

# Sanitize MANIFEST.yaml
cp training/adapters/kimari-smollm3-sft-v0/MANIFEST.yaml /tmp/MANIFEST_staged.yaml
sed -i 's|/workspace/kimari-local-ai/||g' /tmp/MANIFEST_staged.yaml
sed -i 's|/home/[^/]*/kimari-local-ai/||g' /tmp/MANIFEST_staged.yaml
```

### Step 3: Transfer Only Sanitized Files to the Repo Machine

Copy **only** the sanitized artifacts from the training machine to the repo machine. Use a secure, private method (e.g., `scp`, private S3 bucket):

```bash
# From the repo machine, pull only the sanitized files
scp runpod-host:/tmp/MANIFEST_staged.yaml /tmp/
scp runpod-host:eval/results/adapter-smollm3-sft-v0-q4km-summary.json /tmp/
scp runpod-host:eval/results/comparison-sft-v0-vs-baseline-summary.json /tmp/
scp runpod-host:/tmp/private_run_record.json /tmp/

# If screenshots exist, transfer those too (already reviewed per SAFE_SCREENSHOT_CAPTURE.md)
scp runpod-host:docs/assets/screenshots/kimari-*.png /tmp/
```

**What NOT to transfer:**

```bash
# ❌ NEVER transfer these
scp runpod-host:training/adapters/kimari-smollm3-sft-v0/adapter_model.safetensors  # NO
scp runpod-host:training/adapters/kimari-smollm3-sft-v0/checkpoint-*/              # NO
scp runpod-host:models/kimari-*.gguf                                                # NO
scp runpod-host:eval/results/adapter-smollm3-sft-v0-q4km.json                       # NO (raw eval)
```

### Step 4: Place Files in the Repo

Move the sanitized artifacts into their correct locations in the repo:

```bash
# Place the sanitized manifest
# (Adjust destination based on your repo structure and gitignore settings)
cp /tmp/MANIFEST_staged.yaml training/adapters/kimari-smollm3-sft-v0/MANIFEST.yaml

# Eval and compare summaries go in eval/results/
cp /tmp/adapter-smollm3-sft-v0-q4km-summary.json eval/results/
cp /tmp/comparison-sft-v0-vs-baseline-summary.json eval/results/

# Private run record can go in a designated location
cp /tmp/private_run_record.json eval/results/private_run_record.json

# Screenshots go in the screenshots directory
cp /tmp/kimari-*.png docs/assets/screenshots/
```

### Step 5: Run the Pre-Commit Security Scan

Before staging any files, run the secret scanner:

```bash
python scripts/security/scan_for_secrets.py
```

If the scanner reports any hits, **stop and fix** before proceeding. See [Section 6](#6-secret-scanning--run-scan_for_secretspy-before-committing) for details.

### Step 6: Stage and Verify

Stage the files and run verification checks:

```bash
# Stage only the sanitized artifacts
git add training/adapters/kimari-smollm3-sft-v0/MANIFEST.yaml
git add eval/results/adapter-smollm3-sft-v0-q4km-summary.json
git add eval/results/comparison-sft-v0-vs-baseline-summary.json
git add eval/results/private_run_record.json
git add docs/assets/screenshots/kimari-*.png

# Verify staged content
git diff --cached | rg -i 'hf_[a-zA-Z]'        # No HF tokens
git diff --cached | rg '/workspace/'             # No RunPod paths
git diff --cached | rg '/home/[a-z]'             # No home directory paths
git diff --cached | rg '\.safetensors'           # No weight files
git diff --cached | rg '\.bin'                   # No binary files
git diff --cached | rg '\.pt'                    # No PyTorch files
git diff --cached | rg '\.gguf'                  # No GGUF files
git diff --cached | rg 'checkpoint-'             # No checkpoints
git diff --cached | rg '"prompt"'                # No raw prompts
git diff --cached | rg '"response"'              # No raw responses

# Verify only expected files are staged
git diff --cached --name-only
```

**Every `rg` command above must return nothing.** If any command matches, **unstage the offending file and fix the issue.**

### Step 7: Commit

```bash
git commit -m "docs: add sanitized artifacts from private SFT run

- MANIFEST.yaml: sanitized (no absolute paths, gate BLOCKED)
- eval_summary.json: generated by create_eval_summary.py
- compare_summary.json: category-level aggregates only
- private_run_record.json: generated by create_private_run_record.py
- Screenshots: reviewed per SAFE_SCREENSHOT_CAPTURE.md

No weights. No raw outputs. No secrets. Gate stays BLOCKED."
```

## 6. Secret Scanning — Run scan_for_secrets.py Before Committing

The project provides an automated secret scanner at [`scripts/security/scan_for_secrets.py`](../scripts/security/scan_for_secrets.py). **Run it before every commit that includes artifacts from a private training run.**

### What the Scanner Checks

The scanner searches staged and tracked files for patterns that indicate leaked secrets or sensitive information:

| Pattern | Why It Matters |
|---------|---------------|
| `hf_[a-zA-Z0-9]{30,}` | HuggingFace API tokens — can grant repository access |
| `sk-[a-zA-Z0-9]{20,}` | OpenAI-style API keys |
| `ghp_[a-zA-Z0-9]{30,}` | GitHub personal access tokens |
| `/workspace/` | RunPod absolute paths — reveal infrastructure details |
| `/home/[a-z]/` | Home directory paths — may reveal usernames |
| `"prompt"` (in eval JSON) | Raw eval prompts — may contain proprietary instructions |
| `"response"` (in eval JSON) | Raw model responses — may contain real data |
| `.safetensors` | Adapter weight files — proprietary binary data |
| `.gguf` | Quantized model binaries — large, proprietary |

### Running the Scanner

```bash
# Scan all staged files
python scripts/security/scan_for_secrets.py

# Scan specific files
python scripts/security/scan_for_secrets.py --path eval/results/adapter-smollm3-sft-v0-q4km-summary.json

# Scan with verbose output
python scripts/security/scan_for_secrets.py --verbose
```

### If the Scanner Finds Something

1. **Do not commit.** The scanner is a hard gate.
2. Identify the offending file and the specific line.
3. Sanitize the file: remove the secret, replace the path, regenerate the summary.
4. Re-run the scanner to confirm the issue is resolved.
5. Only then proceed to `git add` and `git commit`.

### Manual Redundancy Check

Even with the scanner, always run manual checks for defense in depth:

```bash
git diff --cached | rg -i 'hf_[a-zA-Z]'        # No HF tokens
git diff --cached | rg '/workspace/'             # No RunPod paths
git diff --cached | rg '/home/[a-z]'             # No home directory paths
git diff --cached | rg '\.safetensors'           # No weight files
git diff --cached | rg '\.bin'                   # No binary files
git diff --cached | rg '\.pt'                    # No PyTorch files
git diff --cached | rg 'checkpoint-'             # No checkpoints
git diff --cached | rg '"prompt"'                # No raw prompts
git diff --cached | rg '"response"'              # No raw responses
```

> **Two checks are better than one.** The scanner catches known patterns; manual review catches what the scanner misses.

- [ ] **No weight files** — No `.safetensors`, `.bin`, `.pt`, `.gguf` in any staged file
- [ ] **No checkpoints** — No `checkpoint-*/` directories in any staged file
- [ ] **No raw eval outputs** — No files containing prompt or response text with actual content
- [ ] **No absolute paths** — No `/workspace/`, `/home/`, `/usr/`, or machine-specific paths anywhere
- [ ] **No HF tokens** — No `hf_` strings in any staged file

## 8. Emergency — What to Do If Something Sensitive Was Accidentally Committed

### Scenario A: Committed but NOT Pushed

If you realize the mistake before pushing to the remote:

```bash
# 1. Do NOT push.

# 2. Unstage the offending file
git reset HEAD path/to/sensitive/file

# 3. Amend the commit to remove it
git commit --amend

# 4. Verify the file is no longer in the commit
git show --stat HEAD | rg 'sensitive/file'
```

### Scenario B: Pushed to Remote

If the commit has already been pushed:

1. **Rotate all exposed credentials immediately.**
   - If an HF token was exposed: revoke it at https://huggingface.co/settings/tokens and generate a new one.
   - If any API key was exposed: rotate it through the provider's dashboard.
   - If a password was exposed: change it.

2. **Remove the file from git history.**

   ```bash
   # Using git filter-branch (slow but reliable)
   git filter-branch --force --index-filter \
       'git rm --cached --ignore-unmatch path/to/sensitive/file' \
       --prune-empty HEAD

   # Or using BFG Repo-Cleaner (faster)
   bfg --delete-files path/to/sensitive/file
   git reflog expire --expire=now --all
   git gc --prune=now --aggressive
   ```

3. **Force-push the cleaned history.**

   ```bash
   git push origin --force --all
   git push origin --force --tags
   ```

4. **Notify all collaborators** that history was rewritten and they must re-clone.

5. **Record the incident.** Add a note to the adapter manifest and the run record describing what was leaked, when it was detected, and what remediation was performed.

6. **Consult** [`docs/PRIVATE_RUN_FAILURES.md`](PRIVATE_RUN_FAILURES.md) for additional recovery procedures.

### Scenario C: Sensitive File Was Already Public

If the repository is public and a secret was pushed:

1. **Treat all exposed secrets as compromised.** Rotate them immediately — there are bots that scan public repos for leaked tokens within seconds.
2. **Follow Scenario B steps** to remove the file from history.
3. **Document the incident** in the adapter manifest notes and the project's security log.
4. **Assess the blast radius.** What could the exposed token access? What data was in the raw eval output? Act accordingly.

### Emergency Checklist

- [ ] Stop. Do not push any further commits until the issue is resolved.
- [ ] Rotate all exposed credentials (HF tokens, API keys, passwords).
- [ ] Remove the sensitive file from git history.
- [ ] Force-push the cleaned history.
- [ ] Notify collaborators.
- [ ] Record the incident in the adapter manifest.
- [ ] Review the repository for any other sensitive content that may have been missed.

## 9. Kimari-4B Specific Handoff Notes

The first private SFT run for Kimari-4B has specific considerations beyond the general handoff process above.

### 9.1 What to Bring from the GPU Environment

After completing the Kimari-4B private SFT run, bring **only** these sanitized artifacts:

| Artifact | How to Sanitize | Commit Allowed |
|----------|-----------------|----------------|
| `MANIFEST.yaml` | Remove absolute paths, verify gate BLOCKED | Yes, after review |
| `eval_summary.json` | Generated by `create_eval_summary.py` | Yes, after review |
| `compare_summary.json` | Generated by `compare_runs.py --summary-output` | Yes, after review |
| `private_run_record.json` | Generated by `create_private_run_record.py` | Yes, after review |
| `kimari4b_private_summary.json` | Filled from `training/templates/kimari4b_private_summary.template.json` | Yes, after review |

### 9.2 What NOT to Bring from the GPU Environment

| Artifact | Why It Must Stay Local |
|----------|----------------------|
| `adapter_model.safetensors` | Proprietary adapter weights — never commit |
| `adapter_config.json` (raw) | May contain absolute paths — use sanitized version |
| `checkpoint-*/` | Intermediate training states — never commit |
| `trainer_state.json` (raw) | May contain absolute paths — sanitize before committing |
| `*.gguf` | Quantized model binaries — never commit |
| Base model weights | Downloaded from HF — never commit |
| Raw eval JSON files | Contain full prompts/responses — never commit |

### 9.3 How to Clean Summaries

Before committing any Kimari-4B summary:

1. **Run the secret scanner:**
   ```bash
   python scripts/security/scan_for_secrets.py --paths training eval --json
   ```

2. **Check for absolute paths:**
   ```bash
   rg '/workspace/' training/adapters/kimari4b-smollm3-sft-v0/ 2>/dev/null
   rg '/home/[a-z]' training/adapters/kimari4b-smollm3-sft-v0/ 2>/dev/null
   ```

3. **Verify gate fields:**
   - `preview_gate_state: BLOCKED`
   - `public_release_allowed: false`
   - `hf_upload_allowed: false`

4. **Verify no scores are invented:**
   - All `score_status` fields must be `manual_review_required`
   - No fabricated benchmark numbers
   - No claimed improvements without evidence

### 9.4 Running scan_for_secrets on Kimari-4B Artifacts

```bash
python scripts/security/scan_for_secrets.py \
    --paths training/configs/kimari4b_private_sft_run.v0.yaml \
            training/templates/kimari4b_private_summary.template.json \
            eval/results/ \
            docs/KIMARI4B_PRIVATE_SFT_RUN.md \
    --json
```

If the scanner finds any issues, **do not commit** — fix the issue first.
