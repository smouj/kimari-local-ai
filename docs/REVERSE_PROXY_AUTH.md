# Reverse Proxy Authentication

> **⚠️ Important:** `llama-server` does **NOT** apply the Kimari token natively. The token is intended for use with a reverse proxy or the future Kimari API. Without a reverse proxy enforcing it, the token has no effect.

---

## Architecture

```
client  →  [proxy with auth]  →  Kimari / llama-server on 127.0.0.1:11435
            (nginx / Caddy)         (no built-in auth)
```

- The **client** sends requests with an `Authorization: Bearer <TOKEN>` header.
- The **reverse proxy** validates the token before forwarding.
- **llama-server** accepts all connections from localhost — it has no authentication layer.

---

## ⛔ Do Not Expose llama-server Directly

> **llama-server has no authentication, no rate limiting, and no access control.**
> Anyone who can reach it can use your GPU, send prompts, and consume resources.

**Rules:**

1. **Never** bind llama-server to `0.0.0.0` on a network with untrusted devices.
2. **Never** expose llama-server's port (default `11435`) directly to the internet or an untrusted LAN.
3. **Always** use a reverse proxy (nginx, Caddy, etc.) if you need network access.
4. **Always** bind to `127.0.0.1` (the Kimari default) unless you have a proxy in front.

Kimari configures llama-server to bind to `127.0.0.1` by default. If you override this with `0.0.0.0`, you **must** have a proxy enforcing authentication in front of it.

---

## Kimari Token Does NOT Protect llama-server

The token generated by `kimari token create` is stored locally and is designed to be configured into a reverse proxy. **llama-server itself does not read or validate this token.** If you skip the proxy and connect directly to `127.0.0.1:11435`, the token provides zero protection.

Token enforcement happens **only** in the proxy layer.

---

## Intended Use

- Use **nginx** or **Caddy** as a reverse proxy in front of `llama-server`.
- Kimari binds to `127.0.0.1` (recommended) — the proxy exposes the service to your network.
- **Never** expose `0.0.0.0` publicly without authentication.

---

## Token Setup

Generate and inspect a Bearer token with the CLI:

```bash
# Generate a new token
kimari token create

# Display the current token
kimari token show
```

The token is stored in the user state directory and persists across restarts.

## nginx Example

### With `map` directive for Authorization header

Using `map` is the recommended approach — it avoids the `if` directive inside `server` blocks and integrates cleanly with the authorization logic:

```nginx
# Map the Authorization header to a variable
map $http_authorization $auth_valid {
    default         0;
    "Bearer <YOUR_TOKEN>"  1;
}

server {
    listen 8080;
    server_name _;

    # Reject requests without a valid token
    if ($auth_valid = 0) {
        return 401;
    }

    location / {
        proxy_pass http://127.0.0.1:11435;

        # Standard proxy headers
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE / streaming support
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 86400s;
    }
}
```

> **Note:** Replace `<YOUR_TOKEN>` with the actual token from `kimari token show`.

### Simple `if` approach (alternative)

```nginx
server {
    listen 8080;
    server_name _;

    # Reject anything without the correct Bearer token
    if ($http_authorization != "Bearer <YOUR_TOKEN>") {
        return 401;
    }

    location / {
        proxy_pass http://127.0.0.1:11435;

        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 86400s;
    }
}
```

## Caddy Example

### With `reverse_proxy` and header check

```Caddyfile
:8080 {
    # Token-based auth — reject if Authorization header does not match
    @authnot header Authorization !Bearer <YOUR_TOKEN>
    respond @authnot 401 {
        close
    }

    reverse_proxy 127.0.0.1:11435
}
```

Caddy handles TLS automatically when you use a domain name instead of a bare port:

```Caddyfile
ai.example.com {
    # Token-based auth — reject if Authorization header does not match
    @authnot header Authorization !Bearer <YOUR_TOKEN>
    respond @authnot 401 {
        close
    }

    reverse_proxy 127.0.0.1:11435
}
```

> **Note:** Caddy will provision a certificate via Let's Encrypt automatically. Replace `<YOUR_TOKEN>` with the output of `kimari token show`.

The `close` option in the `respond` directive ensures the connection is terminated immediately on auth failure, preventing any request from reaching the backend.

## Trusted Local Network

If you are on a fully trusted LAN and certain no untrusted device can reach the service, you may skip authentication. However:

- Still bind to `127.0.0.1` and use a proxy to control access.
- **Never** expose `0.0.0.0` publicly without auth.

## Troubleshooting

| Symptom | Likely Cause | Fix |
|---|---|---|
| **401 Unauthorized** | Token mismatch between client and proxy config | Run `kimari token show` and update the proxy config / client header |
| **Connection refused** | Kimari is not running | Run `kimari status` and start the service if needed |
| **502 Bad Gateway** | Proxy pointing to wrong port | Default port is `11435` — verify `proxy_pass` / `reverse_proxy` target |
| **CORS errors** | Browser blocking cross-origin requests | Add `Access-Control-Allow-*` headers in the proxy config |
| **Requests go through without auth** | Proxy not enforcing token check | Verify the `map` / `@authnot` matcher matches your exact token value |
